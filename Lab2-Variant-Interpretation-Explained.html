Part 1

The classic pipeline — data, tools, steps & results

Before any AI, the lab is a standard, deterministic variant-interpretation workflow: parse a VCF, annotate each variant against public knowledgebases, apply ACMG/AMP rules, and classify. Understanding it is the key to understanding what the agent automates.

① What data is used — and is it real?

The lab is built around a sample VCF. A tiny bundled example ships at data/example.vcf — a handful of well-known variants in BRCA1, CFTR and HBB — so it runs even offline. A helper script can additionally pull a small subset of Genome in a Bottle (GIAB) NA12878 / HG001, the gold-standard high-confidence small-variant callset.

Annotation then pulls real, public evidence over the network: MyVariant.info (which aggregates ClinVar clinical significance and gnomAD population allele frequency, plus dbNSFP in-silico scores) and the Ensembl VEP REST API (the most-severe predicted consequence). This is genuine public data — no synthetic fallback is needed beyond the bundled VCF, which itself is real well-characterized biology.

Why this matters scientifically. Unlike a toy dataset, every annotation here is a real community assertion: ClinVar tells you what clinical labs have submitted about a variant, gnomAD tells you how common it is across ~140k human genomes, and VEP tells you what the variant is predicted to do to the protein. The classification you get is only as good as that evidence — which is exactly why the workflow cites its sources for every call.

Step 3b — combining fired criteria into one classification acmg.py · _combine() 
weights = {"stand_alone": 8, "very_strong": 8,
           "strong": 4, "moderate": 2, "supporting": 1}
path_score   = sum(weights[h.strength] for h in hits if h.direction == "pathogenic")
benign_score = sum(weights[h.strength] for h in hits if h.direction == "benign")

if any(h.code == "BA1" for h in hits): return "Benign", ...   # BA1 dominates
if path_score >= 4 and benign_score >= 4:                       # conflicting -> VUS
    return "Uncertain significance (VUS)", ...
if path_score >= 8:   cat = "Pathogenic"
elif path_score >= 4: cat = "Likely pathogenic"
elif benign_score >= 4: cat = "Likely benign"
elif path_score == 0 and benign_score == 0: cat = "Uncertain significance (VUS)"

⑤ The output — and how to read it scientifically

The engine produces one classification per variant. These are the fields that matter:

FieldWhat it meansHow to use it
clinvar_significanceWhat clinical labs have submitted to ClinVar (Pathogenic, Benign, VUS, Conflicting…)Prior community evidence; weigh by review status
gnomad_afPopulation allele frequency across reference genomesRarity — disease alleles are usually very rare; common = likely benign
most_severe_consequenceVEP-predicted effect (e.g. stop_gained, missense_variant)Is it a null/LoF variant? Drives PVS1
criteriaACMG codes that fired — e.g. PVS1, PM2, PP3The evidence trail behind the call
classificationPathogenic / Likely pathogenic / VUS / Likely benign / BenignThe five-tier ACMG verdict
5
classification tiers (Path → Benign)
3
public evidence sources (ClinVar, gnomAD, VEP)
7
heuristic criteria implemented (of 28)

Reading it like a clinical geneticist:

  • ACMG criteria come in categories: pathogenic codes range from PVS1 (very strong — a null variant where loss-of-function causes disease) through PM2 (moderate — rare/absent in gnomAD) to PP3 (supporting — in-silico predictors agree it’s damaging). Benign codes mirror them: BA1 (stand-alone — too common to cause a rare disease), BS1 (strong), BP6 (supporting). Each fired criterion is a line of evidence.
  • What a VUS is: a Variant of Uncertain Significance — the evidence is insufficient or contradictory, so neither “pathogenic” nor “benign” is justified. It is not a diagnosis; it means “we don’t know yet.” Here, no criteria firing, or strong evidence on both sides, both resolve to VUS.
  • Why rare + predicted-damaging + segregation matters: no single line is conclusive. A variant that is rare (PM2) and predicted to break the protein (PP3/PVS1) and tracks with disease in a family (segregation) is far more convincing than any one alone. ACMG’s power is in combining multiple independent lines of evidence, weighted by strength — exactly what _combine() models.
  • Conservative by design: conflicting strong evidence collapses to VUS rather than guessing, and a single common-allele observation (BA1) can outweigh everything else — mirroring real ACMG practice.
Important — simplified, educational subset. This engine implements only a handful of heuristic criteria (roughly PVS1, PM2, PP3, PP5, BA1, BS1, BP6) — not the full ACMG/AMP 2015 framework of 28 criteria with its complete combining-rule table. Real classification needs curated evidence, segregation and functional data, and expert judgment in an accredited lab. This lab is for research / education ONLY — never for clinical diagnosis.

Part 2

The agent — what it adds, how it’s built, how to read its reasoning

The pipeline above runs fine as a plain script. The agent wraps it in an Azure OpenAI model that assembles the evidence, applies the rules consistently, and writes a cited interpretation memo — while being forbidden from inventing numbers or dropping its disclaimer.

Assembles evidence

Calls annotate, then classify, pulling ClinVar / gnomAD / VEP together so a reader sees all the evidence in one place.

Applies rules consistently

Every variant goes through the same ACMG engine in the same order — no ad-hoc judgment, no skipped steps.

Writes a cited memo

Produces a structured markdown memo that lists each fired criterion, cites its sources, and ends with the disclaimer.

① The “skills” the agent possesses

An agent’s skills are the tools it’s allowed to call. This agent has exactly two — the two pipeline functions — plus the built-in ability (granted by its system prompt) to write the final memo.

You are a careful genomics teaching assistant. For the given variant, you
MUST call the annotate_variant tool, then the classify_variant tool, and
only then write a concise markdown interpretation memo. The memo MUST:
(1) state the variant, gene, ClinVar significance, and gnomAD allele
frequency; (2) list each ACMG criterion that fired with its rationale and
the final classification; (3) include a 'Data sources' section citing
MyVariant.info (ClinVar, gnomAD) and Ensembl VEP as applicable; (4) end
with the exact disclaimer provided to you. Never overstate certainty.
Make clear this is a simplified educational classification.

② The code that defines the agent’s behavior

Its engine — the tool-calling loop agent.py · interpret_variant() 
# Bounded loop: the model may call annotate, then classify, then answer.
for _ in range(6):
    resp = client.chat.completions.create(
        model=deployment, messages=messages,
        tools=TOOLS, tool_choice="auto", temperature=0.2)
    msg = resp.choices[0].message
    messages.append(msg.model_dump(exclude_none=True))

    if not msg.tool_calls:                       # no tool -> it's the memo
        content = msg.content or ""
        if "NOT FOR CLINICAL USE" not in content.upper():   # guarantee disclaimer
            content = content.rstrip() + "\n\n" + DISCLAIMER
        return content

    for call in msg.tool_calls:                   # run each requested skill
        args = json.loads(call.function.arguments or "{}")
        result = TOOL_IMPLS[call.function.name](args)
        messages.append({"role": "tool", "tool_call_id": call.id,
                         "name": call.function.name,
                         "content": json.dumps(result, default=str)})

③ How to read the agent’s reasoning output

For each variant the agent prints which tools it called, then the memo it wrote. Learning to skim them is how you audit what the agent did.

Interpreting chr17:g.43045712C>T ...
>> tool call: annotate_variant({"chrom":"17","pos":43045712,"ref":"C","alt":"T"})
>> tool call: classify_variant({"annotation": { ... }})

# Variant Interpretation Memo
## Variant & gene ...  ## ACMG criteria (PVS1, PM2, PP3) ...
## Classification: Likely pathogenic
## Data sources: MyVariant.info (ClinVar, gnomAD), Ensembl VEP
> **DISCLAIMER -- RESEARCH / EDUCATION ONLY. NOT FOR CLINICAL USE.** ...
  • >> tool call: annotate_variant(…) — the agent gathering evidence from the public APIs.
  • >> tool call: classify_variant(…) — the agent running the deterministic ACMG engine.
  • The final # Memo — its written interpretation, saved to output/memos/memo_*.md.
Scientific interpretation — the key habit: every claim in the memo (a fired criterion, a gnomAD frequency, the final class) must trace back to a >> tool call result, not to the model’s imagination. The system prompt forces the annotate→classify→write order, and the loop defensively re-appends the disclaimer if the model ever omits it — so the memo is never presented as clinical advice. To validate, open the annotation JSON and confirm the criteria and class match what the engine returned.

Part 3

From a teaching engine to clinical-grade interpretation

This lab classifies a few variants with seven heuristic rules. Real variant interpretation is far heavier — and that weight is exactly where agentic AI earns its keep.

What “clinical-grade” looks like

  • The full 28 ACMG/AMP criteria plus the Bayesian/points refinements and gene-specific (ClinGen VCEP) rules
  • Curated evidence: segregation, functional studies, case-level and phenotype data — not just automated annotations
  • Backlogs: labs face thousands of variants and a growing mountain of VUS awaiting reclassification
  • Consistency: different curators can reach different calls; reproducibility and concordance are hard
  • Audit & provenance: every classification must be defensible, versioned, and traceable to its evidence

Where the agent pattern scales

  • Evidence assembly — pull ClinVar, gnomAD, VEP, literature and internal cases into one place per variant
  • Consistent rule application — apply the same criteria the same way across every variant, every time
  • VUS triage & re-curation — flag VUS that new population or functional data could now reclassify
  • Cited, audit-ready memos — draft the write-up with every claim traced to a source for human sign-off
  • Natural-language access — “interpret these variants and cite the evidence” instead of bespoke scripts

Value by stakeholder

Clinical labs

Cut the per-variant grind and VUS backlog; the agent assembles evidence and drafts consistent, audit-ready memos — freeing curators for the hard, judgment-heavy cases.

Researchers

Self-serve, cited interpretation across a cohort in plain language, without hand-writing annotation glue or learning every API.

Clinicians

Interpretable, source-cited summaries to support a conversation with a geneticist — provided humans stay in the loop and nothing is used for diagnosis unaided.

The non-negotiable: guardrails

Scaling agentic AI into clinical genetics depends on the same principles this small lab already demonstrates:

  • Evidence-grounded, never fabricated — every criterion and figure traces to a tool result, with the source cited.
  • Human geneticist sign-off — the agent proposes; a qualified molecular geneticist in an accredited (CAP/CLIA) lab reviews and decides.
  • Never autonomous clinical decisions — the mandatory disclaimer is structural, and clinical action requires explicit human approval.
  • Full provenance — annotations, criteria, thresholds and versions are logged so every call is reproducible and defensible.
The big idea: the bottleneck in clinical genomics is interpretation, not sequencing. A geneticist-supervised agent that assembles evidence, applies the ACMG rules consistently, and writes a cited, disclaimer-bearing memo — exactly what you saw in Lab 2 — is how that bottleneck starts to open, from a seven-rule demo to clinical-grade pipelines.

Part 1

The classic pipeline — data, tools, steps & results

Before any AI, the lab is a standard, deterministic RNA-seq differential-expression workflow. Understanding it is the key to understanding what the agent automates.

① What data is used — and is it real?

The lab is built around pasilla, a classic small Drosophila RNA-seq dataset (7 samples, treated vs untreated) distributed with Bioconductor. The downloader tries to fetch it from a public mirror first.

In this run the public URL returned 404 Not Found, so the script automatically fell back to a synthetic counts matrix. That is what produced your results.

Why this matters scientifically. The synthetic generator plants a known ground truth: the first 100 genes are truly up-regulated 3–8× in “treated”, drawn from a negative-binomial distribution so DESeq2 sees realistic dispersion. Everything else is noise. So the “right answer” is ~100 differentially expressed genes — which is exactly why the pipeline recovered 107 significant (≈100 true signals + ~7 expected false discoveries at a 5% false-discovery rate). The biology isn’t real, but it’s a perfect, self-checking teaching dataset.

Code — the planted differential-expression signal _generate_synthetic() 
base_mean = rng.lognormal(mean=3.0, sigma=1.0, size=n_genes)   # per-gene baseline
fc = np.ones(n_genes)
fc[:n_de] = rng.uniform(3.0, 8.0, size=n_de)   # first 100 genes: 3-8x up in 'treated'

def nb_counts(mean_vec):                       # negative-binomial counts
    dispersion = 0.2
    gamma = rng.gamma(1.0/dispersion, mean_vec*dispersion)
    return rng.poisson(gamma)

Reading it like a scientist:

  • Two filters define a “hit”: a large |log2FoldChange| (biologically meaningful change) and a small padj (statistically reliable). A volcano plot puts these on the two axes.
  • Why padj, not pvalue: testing ~2,000 genes at p<0.05 would yield ~100 false positives by chance alone. The FDR adjustment controls that — here it keeps false discoveries to ~5%.
  • Sanity check against ground truth: 107 significant vs 100 planted = the pipeline correctly recovered the signal with a handful of expected false positives. On real data you’d instead validate hits against literature and follow-up experiments.
  • Pathway enrichment then asks “what biology connects these genes?” by testing whether the hit list is over-represented in known pathways (KEGG). On synthetic data this is mechanical; on real data it turns a gene list into a hypothesis.
Honesty note. Because this run used synthetic data, the specific genes and pathways are not real biology — they’re random numbers exercising the exact same machinery. The workflow is 100% real; the findings are placeholders. Drop a genuine counts file into data/counts.csv and the same run yields interpretable results.

Part 2

The agent — what it adds, how it’s built, how to read its reasoning

The pipeline above runs fine as a plain script. The agent wraps it in an Azure OpenAI model that plans the analysis, decides which step to run next, calls the real functions, and writes up the results — while being forbidden from inventing numbers.

Plans & adapts

Chooses the order of steps, reacts to what each result shows, and can skip or retry — instead of a fixed script.

Explains itself

Narrates its reasoning before each action, so a non-specialist can follow why each step happened.

Grounded, not guessing

Every figure it reports comes from a tool result it actually computed — a hard guardrail against fabrication.

The system prompt’s rule “never invent numbers” plus tool-grounding is what makes the agent’s output trustworthy enough for a scientist to act on.

Part 3

From a toy lab to real-world complexity

This lab has four tidy steps. Real bioinformatics is messier — and that mess is exactly where agentic AI earns its keep.

What “complex” looks like

  • Dozens of tools chained across Nextflow / Snakemake, HPC and cloud
  • Multi-omics: DNA, RNA, methylation, proteomics, single-cell, imaging, clinical data
  • Fragile failure modes: wrong genome build, chromosome-naming mismatches, broken sample sheets, silent version drift
  • Branching scientific decisions at every step (thresholds, batch correction, models)
  • Strict reproducibility & provenance demands

Where the agent pattern scales

  • Orchestration of glue work — the same plan→call→check→report loop, over 40 tools instead of 4
  • QC & anomaly detection — catch sample swaps, batch effects, contamination automatically
  • Provenance & reproducibility — record every parameter, version and input as it goes
  • Literature-to-analysis — connect dataset findings to published evidence, with citations
  • Natural-language access — ask a question in English instead of writing pipeline code

Value by stakeholder

Bioinformaticians

Less time on brittle glue and re-runs; the agent handles orchestration, logs, and first-pass QC — freeing experts for study design and edge cases.

Researchers

Self-serve analysis in plain language (“compare treated vs untreated and show enriched pathways”) without waiting on a queue or learning the toolchain.

Clinicians

Interpretable, cited, governed summaries — provided humans stay in the loop and outputs are validated, never used for diagnosis unaided.

The non-negotiable: guardrails

Scaling agentic AI safely depends on the same principles this small lab already demonstrates:

  • Tool-grounded, never fabricated — claims trace to computed results, with artifacts to verify them.
  • Human-in-the-loop — the agent proposes and executes; a qualified person reviews and decides.
  • Full provenance — every step, parameter and version is logged and reproducible.
  • Bounded autonomy — clinical or wet-lab actions require validation and explicit approval.
The big idea: the bottleneck in modern genomics is interpretation and orchestration, not sequencing. A scientist-supervised agent that plans, runs, explains and documents analyses — exactly what you saw in Lab 1 — is how that bottleneck starts to open, from a 4-step demo to production pipelines.

Advanced

Multi-omics tumor board — where the agent chooses the tools

A harder sequel to the RNA-seq lab. Instead of one fixed 4-step pipeline, the agent has 12 tools across four omics layers of a human breast-cancer case and must decide which to run based on what it finds — a conditional, non-sequential workup, like a molecular tumor board.

Why it's harder. Lab 1's pipeline always runs 1→2→3→4. Here the path changes per case: the agent only confirms HER2 amplification if HER2 looks high, only profiles immunity if mutational burden is high or BRCA is hit, and skips a whole omics layer when it's missing. The order of skills is an outcome of the data, not a script.

⑤ The output, and how to read it

Each run prints the agent's reasoning, the TOOL PATH it took, and a final “Molecular Tumor Board Summary” with: case & subtype, key molecular findings (each citing a tool + value), the tools it skipped and why, and a biomarker → therapy-class hint.

  • Read the skips, not just the runs — the value is the agent justifying what it didn't do (e.g. “skipped immune profiling: TMB low and no BRCA”).
  • Every finding is tool-grounded — a “TMB = 14 mut/Mb” in the report traces to a mutation_analysis result, never invented.
  • The therapy hint is a pointer, not a prescription — it maps a biomarker to a drug class for learning only.
Why this matters for real pipelines: production bioinformatics is exactly this — dozens of tools, branching decisions, optional or missing data layers. An agent that chooses tools by condition, explains each choice, and grounds every number is how the pattern scales beyond a fixed script — with a human tumor board still making the call.
Research / education only. Synthetic or public research data and simplified heuristics. Not for diagnosis or treatment — real molecular tumor boards require board-certified clinicians and accredited (CAP/CLIA) testing.

The classic pipeline — data, tools, steps & results

Before any AI, the lab is a standard, deterministic variant-interpretation workflow: parse a VCF, annotate each variant against public knowledgebases, apply ACMG/AMP rules, and classify. Understanding it is the key to understanding what the agent automates.

① What data is used — and is it real?

The lab is built around a sample VCF. A tiny bundled example ships at data/example.vcf — a handful of well-known variants in BRCA1, CFTR and HBB — so it runs even offline. A helper script can additionally pull a small subset of Genome in a Bottle (GIAB) NA12878 / HG001, the gold-standard high-confidence small-variant callset.

Annotation then pulls real, public evidence over the network: MyVariant.info (which aggregates ClinVar clinical significance and gnomAD population allele frequency, plus dbNSFP in-silico scores) and the Ensembl VEP REST API (the most-severe predicted consequence). This is genuine public data — no synthetic fallback is needed beyond the bundled VCF, which itself is real well-characterized biology.

Why this matters scientifically. Unlike a toy dataset, every annotation here is a real community assertion: ClinVar tells you what clinical labs have submitted about a variant, gnomAD tells you how common it is across ~140k human genomes, and VEP tells you what the variant is predicted to do to the protein. The classification you get is only as good as that evidence — which is exactly why the workflow cites its sources for every call.

    (cadd is not None and cadd >= PP3_CADD_MIN)
    or sift.startswith("D") or polyphen.startswith(("D", "P")))
Step 3b — combining fired criteria into one classification acmg.py · _combine() 
weights = {"stand_alone": 8, "very_strong": 8,
           "strong": 4, "moderate": 2, "supporting": 1}
path_score   = sum(weights[h.strength] for h in hits if h.direction == "pathogenic")
benign_score = sum(weights[h.strength] for h in hits if h.direction == "benign")

if any(h.code == "BA1" for h in hits): return "Benign", ...   # BA1 dominates
if path_score >= 4 and benign_score >= 4:                       # conflicting -> VUS
    return "Uncertain significance (VUS)", ...
if path_score >= 8:   cat = "Pathogenic"
elif path_score >= 4: cat = "Likely pathogenic"
elif benign_score >= 4: cat = "Likely benign"

Reading it like a clinical geneticist:

  • ACMG criteria come in categories: pathogenic codes range from PVS1 (very strong — a null variant where loss-of-function causes disease) through PM2 (moderate — rare/absent in gnomAD) to PP3 (supporting — in-silico predictors agree it’s damaging). Benign codes mirror them: BA1 (stand-alone — too common to cause a rare disease), BS1 (strong), BP6 (supporting). Each fired criterion is a line of evidence.
  • What a VUS is: a Variant of Uncertain Significance — the evidence is insufficient or contradictory, so neither “pathogenic” nor “benign” is justified. It is not a diagnosis; it means “we don’t know yet.” Here, no criteria firing, or strong evidence on both sides, both resolve to VUS.
  • Why rare + predicted-damaging + segregation matters: no single line is conclusive. A variant that is rare (PM2) and predicted to break the protein (PP3/PVS1) and tracks with disease in a family (segregation) is far more convincing than any one alone. ACMG’s power is in combining multiple independent lines of evidence, weighted by strength — exactly what _combine() models.
  • Conservative by design: conflicting strong evidence collapses to VUS rather than guessing, and a single common-allele observation (BA1) can outweigh everything else — mirroring real ACMG practice.
Important — simplified, educational subset. This engine implements only a handful of heuristic criteria (roughly PVS1, PM2, PP3, PP5, BA1, BS1, BP6) — not the full ACMG/AMP 2015 framework of 28 criteria with its complete combining-rule table. Real classification needs curated evidence, segregation and functional data, and expert judgment in an accredited lab. This lab is for research / education ONLY — never for clinical diagnosis.

① What data is used — and is it real?

The lab is built around PBMC 3k, the classic Scanpy tutorial dataset: real single-cell RNA-seq of ~2,700 peripheral blood mononuclear cells from a healthy donor, sequenced on the 10x Genomics platform (Cell Ranger 1.1.0).

It is loaded directly via scanpy.datasets.pbmc3k(), which fetches the filtered gene-barcode matrix and returns a ready-to-use AnnData object. If the loader can't reach the internet, the script falls back to downloading the raw 10x .tar.gz and parsing the MTX triplet.

What the matrix actually is. Single-cell data is a genes × cells count matrix: each column is one cell, each row is one gene, each entry is how many transcripts of that gene were captured in that cell. It is overwhelmingly sparse — most genes are zero in most cells — because only a few thousand of the ~20,000 genes are detectably expressed per cell. This is real biology, not synthetic: the cells, the donor, and the counts are genuine, which is exactly why PBMC 3k is the reference dataset for learning the workflow.

Reading it like a scientist:

  • Clusters are not cell types — yet. Leiden returns numbered groups. They only become biology once a marker fingerprint is matched to a canonical cell identity.
  • Resolution is a judgment call. LEIDEN_RESOLUTION directly controls how many clusters you get — too low merges distinct types, too high splits one type into artificial fragments. There is no single "correct" number.

Honesty note. The data is real, but the labels are inferred: a cluster called "NK cells" is a hypothesis supported by its markers, not a measured fact. The workflow is canonical; the annotations still require expert validation before they drive any biological conclusion.

① What data is used — and is it real?

The lab references real public Perturb-seq / CRISPR screens — Norman 2019, Adamson 2016, and the genome-wide Replogle 2022 — as the kind of readout you would wire in for production. To keep the active-learning loop runnable entirely offline, it ships with a pathway-clustered synthetic screen simulator that provides a known ground-truth gene→phenotype table.

A CRISPR knockout screen measures the phenotypic effect of disabling each gene one at a time; the simulator stands in for that assay by returning a noisy phenotype score per gene. Because the truth is known, the agent's choices can be scored objectively without any wet-lab work.

Why this matters scientifically. The simulator plants a known ground truth: genes are organized into pathway clusters, and two pathways (MAPK_proliferation and cell_cycle) are designated strong-hit pathways centered near an effect of 0.85, while the rest sit near 0.05. A gene counts as a "hit" when its true effect exceeds 0.5. This structure rewards an agent that, after finding one hit, hypothesizes that its pathway neighbors are also hits — the exact structure active learning should exploit.

def measure(self, gene: str) -> float:
    # Simulate one screen readout: true effect + Gaussian noise.
    base = self.truth.get(gene, 0.0)                 # hidden true phenotype effect

Reading it like a scientist:

  • Sample-efficiency is the metric: the agent wins not by finding more hits in the limit, but by finding them in fewer experiments — the same hits, sooner. On a real screen that is saved reagents, time and money.
  • Why the agent beats random: once a hit in MAPK_proliferation shows up, the critic steers the next batch toward untested members of that same pathway, harvesting a cluster of hits at once instead of sampling blindly.
  • Noise is realistic: because measure() adds Gaussian noise, a single readout can mislead — which is why repeated measurements and pathway-level evidence matter, just like biological replicates.
